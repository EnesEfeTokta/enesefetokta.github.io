Building a Microservices Architecture with .NET 8

Enes Efe Tokta

Jan 15, 2026 • 8 min read

.NET 8 Microservices Docker Architecture

Microservices architecture has become the de facto standard for building scalable, maintainable enterprise applications. In this comprehensive guide, I'll walk you through the process of designing and implementing a microservices system using .NET 8, sharing insights from my experience building production-ready distributed systems.

Why Microservices?

Before diving into implementation, it's crucial to understand when microservices make sense. While they offer numerous benefits, they also introduce complexity. Here are scenarios where microservices excel:

  • Independent Scaling: Different parts of your application have varying resource requirements
  • Team Autonomy: Multiple teams working on different features without stepping on each other's toes
  • Technology Flexibility: Freedom to choose the best technology stack for each service
  • Fault Isolation: Failures in one service don't bring down the entire system

Pro Tip

Don't start with microservices if you're building an MVP or a small application. The operational complexity isn't worth it. Start monolithic and migrate to microservices when you have clear domain boundaries and scaling needs.

Core Principles of Microservices

1. Service Independence

Each microservice should be independently deployable and have its own database. This principle, known as database per service, ensures that services don't share data stores, preventing tight coupling. 

// Example: Service structure in .NET 8
services/
├── UserService/
│   ├── UserService.API/
│   ├── UserService.Domain/
│   └── UserService.Infrastructure/
├── ProductService/
│   ├── ProductService.API/
│   ├── ProductService.Domain/
│   └── ProductService.Infrastructure/
└── OrderService/
    ├── OrderService.API/
    ├── OrderService.Domain/
    └── OrderService.Infrastructure/

2. API Gateway Pattern

An API Gateway serves as the single entry point for clients, routing requests to appropriate microservices. In .NET, we can use Ocelot or YARP (Yet Another Reverse Proxy) for this purpose. 

// Ocelot configuration example
{
  "Routes": [
    {
      "DownstreamPathTemplate": "/api/users/{everything}",
      "DownstreamScheme": "http",
      "DownstreamHostAndPorts": [
        {
          "Host": "user-service",
          "Port": 5001
        }
      ],
      "UpstreamPathTemplate": "/users/{everything}",
      "UpstreamHttpMethod": [ "GET", "POST", "PUT", "DELETE" ]
    }
  ]
}

3. Service Communication

Microservices need to communicate with each other. There are two main patterns:

  • Synchronous (HTTP/gRPC): Direct request-response communication
  • Asynchronous (Message Queue): Event-driven communication using RabbitMQ, Azure Service Bus, or Kafka
"In my experience, a hybrid approach works best. Use synchronous communication for queries and asynchronous messaging for commands and events."

Implementation Example: Order Processing System

Let's build a simple order processing system with three microservices: User Service, Product Service, and Order Service.

Setting Up the Project

// Creating a new microservice with .NET CLI
dotnet new webapi -n OrderService.API
dotnet new classlib -n OrderService.Domain
dotnet new classlib -n OrderService.Infrastructure

// Add necessary packages
cd OrderService.API
dotnet add package Microsoft.EntityFrameworkCore
dotnet add package MassTransit.RabbitMQ
dotnet add package Serilog.AspNetCore

Domain Model Example

// OrderService.Domain/Entities/Order.cs
public class Order
{
    public Guid Id { get; set; }
    public Guid UserId { get; set; }
    public List<OrderItem> Items { get; set; } = new();
    public decimal TotalAmount { get; set; }
    public OrderStatus Status { get; set; }
    public DateTime CreatedAt { get; set; }
    
    public void CalculateTotal()
    {
        TotalAmount = Items.Sum(item => item.Price * item.Quantity);
    }
}

public enum OrderStatus
{
    Pending,
    Confirmed,
    Shipping,
    Delivered,
    Cancelled
}

Event-Driven Communication

// Publishing an event when order is created
public async Task<IActionResult> CreateOrder([FromBody] CreateOrderDto dto)
{
    var order = new Order
    {
        Id = Guid.NewGuid(),
        UserId = dto.UserId,
        Items = dto.Items,
        Status = OrderStatus.Pending,
        CreatedAt = DateTime.UtcNow
    };
    
    order.CalculateTotal();
    await _orderRepository.AddAsync(order);
    
    // Publish event
    await _publishEndpoint.Publish(new OrderCreatedEvent
    {
        OrderId = order.Id,
        UserId = order.UserId,
        TotalAmount = order.TotalAmount
    });
    
    return Ok(order);
}

Containerization with Docker

Docker is essential for microservices. Here's a production-ready Dockerfile for our .NET service: 

# Dockerfile
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS base
WORKDIR /app
EXPOSE 80

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY ["OrderService.API/OrderService.API.csproj", "OrderService.API/"]
RUN dotnet restore "OrderService.API/OrderService.API.csproj"
COPY . .
WORKDIR "/src/OrderService.API"
RUN dotnet build "OrderService.API.csproj" -c Release -o /app/build

FROM build AS publish
RUN dotnet publish "OrderService.API.csproj" -c Release -o /app/publish

FROM base AS final
WORKDIR /app
COPY --from=publish /app/publish .
ENTRYPOINT ["dotnet", "OrderService.API.dll"]

Best Practices & Lessons Learned

Key Takeaways

  1. Start Simple: Don't over-engineer from day one. Add complexity as needed.
  2. Observability is Crucial: Implement logging, monitoring, and distributed tracing from the start (Serilog, OpenTelemetry).
  3. Health Checks: Every service should expose health check endpoints for orchestrators like Kubernetes.
  4. Resilience Patterns: Implement Circuit Breaker, Retry, and Timeout patterns using Polly.
  5. API Versioning: Plan for API evolution from the beginning.

Common Pitfalls to Avoid

  • Distributed Monolith: Creating many services that are still tightly coupled
  • Network Chattiness: Too many inter-service calls degrading performance
  • Inconsistent Data: Not handling eventual consistency properly
  • Missing Service Discovery: Hardcoding service URLs instead of using service discovery

Conclusion

Building microservices with .NET 8 is powerful but requires careful planning and discipline. Focus on domain boundaries, embrace asynchronous communication where appropriate, and invest in observability from the start.

Remember: microservices are a means to an end, not the end itself. The goal is to build maintainable, scalable systems that deliver business value.
